Appendix

How the pricing audit works

This page explains the methodology behind the dashboard in plain language and mirrors the actual rules implemented in the local audit app.

Back to dashboard Shared conversation

Core idea

Compare PriceLabs to a weighted local comp set.

The goal is not to reverse-engineer PriceLabs perfectly. It is to test whether the recommendation looks defensible versus nearby, similar, bookable inventory.

The bundled Zalor Beach example is synthetic. The workflow is real, but the sample numbers are only a scaffold until you replace them with actual PriceLabs exports and observed competitor calendar prices.

Inputs

What data goes into the audit

`subjectListing` The listing we are judging: bedrooms, guest count, pool, walk time, rating, and notes.
`pricelabsRates` The nightly prices that PriceLabs recommended for the forward window.
`comparables` Observed market listings with attributes plus daily prices, availability flags, and optionally a `listingUrl` for the exact OTA page.
Public PriceLabs pages Scraped only for context. They are not used as benchmark data for the nightly pricing calculations.

Workflow

High-level sequence

  1. Normalize the subject listing, PriceLabs rate series, and comp calendars into a common date-keyed structure.
  2. Filter the comp set using the UI controls: horizon, minimum rating, max walk time, pool strictness, and minimum similarity.
  3. Score each surviving comparable for similarity to the subject listing.
  4. For each day, compute a weighted market median and band from the available comps.
  5. Compare the PriceLabs nightly price against that benchmark and classify the day as `Raise`, `Hold`, or `Lower`.
  6. Aggregate daily gaps into summary metrics like median gap, price index, revenue uplift, and confidence.

Similarity

How comparable matching is scored

The app computes a similarity score for each candidate comp using five components. Those components are weighted, then summed into a single score between `0` and `1`.

bedroomScore = max(0, 1 - abs(subject.bedrooms - comp.bedrooms) / 3) guestScore = max(0, 1 - abs(subject.maxGuests - comp.maxGuests) / 6) distanceScore = max(0, 1 - abs(subject.walkMinutesToBeach - comp.walkMinutesToBeach) / 20) ratingScore = max(0.55, comp.rating / 5) poolScore = 1 when pool setup matches 0 when strict pool matching is on and it does not match 0.7 when strict pool matching is off and it does not match
similarity = bedroomScore * 0.30 + guestScore * 0.18 + poolScore * 0.20 + distanceScore* 0.14 + ratingScore * 0.18

Only comps above the chosen `minimum similarity` threshold make it into the benchmark set.

Benchmark

How the market price is derived

For each date, the app looks at only the comparable listings that:

  • survive the filtering rules,
  • have a rate for that date, and
  • are marked as `available` on that date.

Each usable comp gets a daily weight based on similarity and rating. The market benchmark is then calculated using weighted quantiles:

dailyWeight = max(0.05, similarity * (0.7 + rating / 10)) marketLow = weightedQuantile(0.25) marketMedian = weightedQuantile(0.50) marketHigh = weightedQuantile(0.75)

This produces both a centerline and a plausible market band instead of pretending there is only one correct number.

Decision rules

How the daily recommendation is classified

gapAmount = marketMedian - priceLabs gapPercent = (gapAmount / marketMedian) * 100 priceIndex = priceLabs / marketMedian
`Raise` When `gapPercent >= 8`. PriceLabs is materially below the market median.
`Hold` When the daily gap is inside the `-8%` to `+8%` band.
`Lower` When `gapPercent <= -8`. PriceLabs is materially above the market median.
`Need more data` When too few usable comps remain for that date after filtering.

Confidence

How confidence and occupancy proxy are estimated

The confidence score is not a statistical confidence interval. It is a practical quality score driven by comp count and comp quality.

confidence = min(1, (usableComparableCount / 5) * (averageSimilarity + 0.35)) * 100

The occupancy proxy comes from availability flags on the comp calendars, not from actual booked revenue data.

marketOccupancyProxy = unavailableComparableDates / coveredComparableDates * 100

This gives a quick demand signal, but it should be treated as directional rather than exact.

Summary layer

How the top-line verdict is produced

Once all daily rows are computed, the app rolls them up into a single summary:

  • `medianPriceLabs` and `medianMarket` are the medians of the daily series.
  • `medianGapPercent` compares those two medians.
  • `opportunityUplift` sums only positive daily gaps, showing modeled upside if underpriced days move to the market median.
  • `downsideRisk` sums only negative daily gaps, showing modeled overpricing exposure.
`Likely underpriced` When the rolled-up median gap is at least `+8%`.
`Within market band` When the rolled-up median gap stays between `-8%` and `+8%`.
`Likely overpriced` When the rolled-up median gap is at most `-8%`.

Limits

What this appendix is not claiming

  • The app does not scrape private PriceLabs app APIs or reverse-engineer their internal model.
  • The app does not account for cleaning fees, cancellation rules, host responsiveness, photos, or conversion funnel effects.
  • Seasonality matters only to the extent that it is already visible in the comp-set rates you provide.
  • The bundled sample is illustrative, not investment-grade evidence.

Real data

How to make the audit meaningful

  1. Export or copy actual `PriceLabs` daily recommendations for the forward window you care about.
  2. Capture comparable listing calendars from Airbnb, Vrbo, or your market dashboard workflow. The dashboard can now generate Google search links for each comparable, and it will use a direct `listingUrl` if you add one to the JSON.
  3. Replace the sample JSON with those real observations in the dashboard editor.
  4. Rerun the audit and adjust the controls until the comp set reflects the exact micro-market you trust.

Automation

What can be automated next

  • Discovery can be automated now by generating Google search links from listing name, source, and micro-market.
  • Accuracy improves if you save the canonical OTA page into each comparable as `listingUrl` and reuse it in future refreshes.
  • The next practical automation step is browser automation that opens those known listing pages and captures visible prices and availability into JSON.
  • Google result scraping itself is fragile, so the safer workflow is: search once, save the listing URL, then automate from that stable source.
